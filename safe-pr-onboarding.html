safe-pr · claude-non-dev-pr

Ship a small change safely — without writing code.

A Claude Code plugin that walks a non-developer — a UX designer, PM, or CS teammate — from an idea to a draft pull request a developer reviews. It runs a deterministic safety gate plus a risk assessment, and ships only what it can prove is small, bounded, and checkable by hand.

v0.12.1 Node ≥ 20 deps: zero runtime tests: node:test needs: git · gh 14-step guided flow

What it is

A filter to reduce review burden — not a merge wall.

safe-pr guides the operator (the non-developer) in plain language and does the technical work for them — asking only for the decisions a human must own: what the change should be, and whether it came out right. The real backstops stay exactly where they were: the repo's own CI and a developer reviewing every PR. Everything runs in-session — no custom CI check, no repo scaffolding to install.

Invariant 1

Bounded blast radius

The change is confined to a deny-list-clean, small, mappable surface — enforced by the gate's file/LOC ceilings and a consumer analysis of everything it can reach.

Invariant 2

Hand-verifiable

The operator can observe both that the change worked and that nothing else moved — and the failure modes are visible, not silent. A change qualifies only when it clears both. The rest correctly hand off — that's a success, not a gap.

How it works · two layers of truth

A deterministic gate, checked by reasoning.

Every safety decision is split between a script that always returns the same answer for the same input, and the model's judgment fed by that script's facts. The rule that keeps them honest: evidence before score — never state a rating until the facts that justify it are written down, each with the command that produced it.

■ Layer A — deterministic

bin/pr-gate.js

Plain JavaScript on Node built-ins. Emits machine-readable JSON that is ground truth — the model quotes it and never recomputes the numbers by eye.

  • preflight — clean tree, branch & base, gh readiness
  • new-branch — validate the Jira id, build the branch name
  • check-paths — deny-list pre-check before editing
  • scan-diff — deny hits, file/LOC ceilings
  • ci-status — stability-aware CI read

hard_fail: true (deny hit or over ceiling) is an automatic STOP.

▢ Layer B — reasoning

The model's assessment

Scores the change on three axes using Layer A's facts, plus judgment the script can't make: what a change can actually reach, and whether a human could catch it going wrong.

  • Risk / Complexity / Confidence, each Red · Yellow · Green
  • Every rating cites the grep, file read, or JSON that backs it
  • Maps the blast radius to a regression checklist
  • Decides whether the repo convention expects a test

Reasoning never overrides Layer A — a Red can't be talked past.

The verdict · one form only

Three axes, three lights.

There is no separate "in scope / out of scope" call — every decision goes through this scorecard. Score the specific change, not the category it falls in or how central the file is.

Axis Green Yellow Red
Riskblast radius
 GreenLeaf / presentational; ≤ 1–2 consumers; not on a critical path. YellowShared by several consumers, or a contained state / prop change. RedShared logic consumers depend on, a money / auth / data path, or an unbounded surface.
Complexitysize & shape
 Green1–2 files, presentational only, no logic. YellowA few files, minor localized logic, or a contained default change. RedRippling logic / timing, data fetching, a new component or pattern.
Confidencesure nothing breaks
 GreenGood coverage & green CI — or fully hand-verifiable and confirmed end-to-end. YellowThin coverage or 1–2 tests adjusted, but failure modes stay observable. RedSilent failure modes, many tests adjusted, CI failing, or couldn't verify.

Aggregation: worst of the three

The final verdict is the worst single axis. Two or more yellows stay Yellow — they do not compound to Red. Yellow proceeds (flagged for closer review); Red stops and hands off.

Two proxies to distrust

File centrality describes the file, not the change — a one-line default in a central file has a small blast radius. And the word "state" decides nothing: a stored default is contained; a data-fetch controller ripples.

Scoring in practice

Same word, different verdict.

Three changes that all touch "state" or "spacing" — and land in three different places, because scope and verifiability decide, not the label.

● Green · ship it

Rename one button "Subscribe" → "Join now" on the pricing page.

Why: one importer, pure copy, snapshot test covers it, verified in the running app.

● Yellow · ship, flagged

Remember the last-open tab via one sessionStorage key in the tab's initial value.

Why: the context is shared, but this touches only the initial value — contained, reversible, and fully visible on screen.

● Red · hand off

Restore scroll position when navigating back from a details page to the list.

Why: it hooks into scroll + data-load timing other flows depend on — rippling logic that fails silently on a slow load.

The allowed-change catalog

What may be shipped — and how it's checked.

Inside the deny-list fence, three change natures qualify. The model names the section at the plan gate; that choice decides the verification method.

Presentational

Copy & text, colors, spacing, sizing, simple style props, icon swaps, toggling a class.

Verify by Running the app and looking at the surface — the effect, and that nothing else moved.

Content

Docs & markdown, help articles, message strings, i18n string values, in-repo canned replies. Not anything flag- or config-driven.

Verify by Rendering the output and reading it — trigger the state that shows the string and read it in context.

Bounded change

One localized, self-contained, reversible default / prop / state change that can't reach the file's other responsibilities.

Verify by Exercising the exact path by hand — confirm the effect and that nothing else moved.

The deterministic fence

Some files stop the run before it starts.

Touching any of these is an automatic STOP — no reasoning, straight to a developer handoff. The gate enforces it (deny_hits), and the model refuses to even start down the path. Narrow scope is the safety argument.

Developer handoff

A clean stop, not a dead end

  1. The confirmed spec + acceptance criteria.
  2. What was attempted and the diff so far.
  3. The blocking evidence — which axis went Red, or the Layer A hard_fail.
  4. A plain note to the operator: what happened, that it's not their fault, and that a developer takes it from here.

For developers · extending the plugin

Pure logic behind I/O seams.

The gate is plain JavaScript on Node built-ins — no runtime dependencies. Decision logic lives in lib/ behind injected git/gh seams so it stays unit-testable; bin/pr-gate.js is the thin CLI that wires real I/O in. Keep that split, and keep "same input → same output."

Ground rules
01

Deterministic gate is ground truth. Changes to lib//bin/ keep same-input→same-output. Don't move risk decisions between the script and the skill.

02

No runtime dependencies. Node built-ins only; never add a manifest with runtime deps. Run the suite with node --test.

03

Config is a fence, not a risk map. gate.config.json holds deny-list globs and hard limits — max_files: 8, max_loc: 150, max_test_loc: 150 (product and test LOC budgeted separately). A repo drops .safe-pr.config.json to override.

04

Flow steps are whole integers. To insert a step, renumber the rest — never a fractional label. Refer to steps by name in prose, not by number.

Versioning — required on every plugin change

Pick the level by change type (semver):

patch

Bug fixes, wording, docs, test-only, threshold tuning. No contract change.

minor

New backward-compatible behavior — a new gate, subcommand, or config option with a safe default.

major

Breaking — removing / renaming a subcommand or key, or changing the gate's JSON contract.

Two files must stay in sync. Bump the version in plugins/safe-pr/.claude-plugin/plugin.json and the plugin's entry in .claude-plugin/marketplace.json together — never one without the other.

Get started

Install & run

# add the marketplace (owner/repo, git URL, or local path)
/plugin marketplace add <owner>/<repo>
/plugin install safe-pr@claude-non-dev-pr
# then, in a target repo:
/safe-pr:safe-pr

safe-pr is a skill invoked directly — there is no command wrapper (a same-named command would shadow the skill and stop its body from loading). Each target repo needs a top-level CLAUDE.md with build/run instructions and tests that run in CI per PR.
